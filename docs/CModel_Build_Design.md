# CModel 从零构建设计规范

基于全周期解析所取得的经验和 100% 精确的标签定位，现在得出从零重新构建、装配并导出完全合规且无数据损坏的 `.cmodel` 文件的详细规范。在后续重构（Phase 4）中，后端引擎必须完全遵照本规范重写序列化流程。

## 1. 架构选型与构建策略：Dictionary to Protobuf
过往系统的最大设计败笔在于：试图直接操控底层或使用了无法精确控制数据类型的 JSON 转换套件，导致了极为严重的 IEEE 754 浮点数污染与乱码。

**全新构建策略**：
1. **基于 Python 原生 Dictionary**：在内存中完全按标签构建嵌套的 `dict` 与 `list`。
2. **利用 `blackboxprotobuf.encode_message` 刚性约束**：引入提取出的绝对 Schema 标准大纲 `CompDesc.model`，向其直接注入原生的 Python `float`, `bytes`, `int`。使得序列化引擎严格将其按规定的 C++ 底层原语编码为 Protobuf 字节流。

## 2. 数据流与层级装配向导

### 2.1 Root 容器级装配
创建一个顶层 `Payload` 字典：
```python
payload = {
    "5": []  # Module Node Array
}
```

### 2.2 定义 Module Node
向 `payload['5']` 不断 `append` 新的模块字典。
一个标准的 Module Node 字典应包含：
```python
node = {
    "5": generate_uuid().encode('utf-8'), # 必须是 bytes 格式的 UUID
    "6": b"driver",                       # 模块类型 (bytes)
    "7": b"driver-left",                  # 模块实称 (bytes)
    "8": b"120x60x25",                    # 三维尺寸
    "4": [                                # 核心配置容器
        {
           "1": [], # relations (空间变换与从属)
           "2": {"1": []}, # private attributes (私有物理电控属性)
           "3": []  # interfaces (总线与接口)
        }
    ]
}
```
**注意：Protobuf 的字符串本质是长度前缀的字节数组，因此所有中文字符和英文字符串在推入字典前，必须经过 `.encode("utf-8")` 转换为 `bytes`。这彻底根治乱码。**

### 2.3 生成空间拓扑关系 `Node['4'][0]['1']`
该列表描述模块的父子树。
如果要挂载到主底盘对象 `chassis_diff`：
```python
relation = {
    "4": parent_uuid_bytes,
    "5": b"P",           # P代表Parent
    "6": float_x,        # Python 原生 float
    "7": float_y,
    "8": float_z,
    "9": float_pitch,    # 原生 float...
    "10": float_yaw,
    "11": float_roll
}
# 推入 Node['4'][0]['1'].append(relation)
```

### 2.4 组装私有属性 `Node['4'][0]['2']['1']`
在 `2.1` 下可追加多个 group 字典。必须严格遵循“数组包字典”或者如果单组就放单字典。
```python
prop_group = {
    "2": b"底盘参数",  # 组名
    "3": []
}
prop_group["3"].append({
    "51": "最大线加速度（空载）".encode('utf-8'),
    "2": 10, # 代表这是一个浮点类型
    "17": 500.0, # 此处由于我们使用 schema 序列化，底层可直接使用 float / int！黑盒工具会自动处理。
})
```
*备注：根据前面解析脚本的表现，我们知道不同的 `datatype` 对应不同的标签号，在装配前必须实现一个简单的 `datatype_to_tag` 的路由映射器*

### 2.5 建立完美闭环电气连接网络
最容易出错的一步是被挂接端口由于拼写错误或者 UUID 丢失导致的解析断层。
构建流程需分为“组件初始化”和“网络布线”两步走。
1. **实例化接口（组件初始化）**：创建所有模块的接口描述。强制为每一个接口生成一个全新的随机 `UUID`。存储在 `port['5']`。
2. **连接绑定（网络布线）**：
   假设我们要把 A 模块的 `CAN_1` 连向 B 模块的 `CAN_2`：
   后端必须要有一个“中间布线引擎”，获取 A 的 `CAN_1` 的 `UUID_A`，和 B 的 `CAN_2` 的 `UUID_B`。
   然后，将 `UUID_B` 存入 `A_CAN_1['6']` 这个列表里。
   同理（如果是双向映射），将 `UUID_A` 存入 `B_CAN_2['6']` 列表里。
   以此达到逻辑上的 100% 精准闭环寻址，舍弃所有原系统靠字符串硬拼的玄学作法。

## 3. 持久化打包为 `.cmodel`
当所有的节点如上述被注册进 `payload` 之后：
1. **编译写入**：
```python
with open("templates/CompDesc.model", "rb") as f:
    schema = list(blackboxprotobuf.lib.types.generate_typedefs(f.read()).values())[0]

# 严格按 Schema 进行序列化
binary_data = blackboxprotobuf.encode_message(payload, schema)
with open("Model.model", "wb") as f:
    f.write(binary_data)
```
2. **打包生成**：
编写 JSON：
```json
{
    "projectId": "1",
    "projectName": "New AMR",
    "createTime": "2026-03-XX ...",
    "description": "Generated by V2 Extractor Engine"
}
```
最后使用 Python 内置的 `zipfile` 库，以标准 ZIP 算法将 `baseInfo.json` 与 `.model` 打在一起，并将后缀重命名为 `.cmodel`。导出，工作结束！
